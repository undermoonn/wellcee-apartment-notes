# 筛选与管理 Wellcee 房源

使用 Chrome 按用户条件评估 Wellcee 房源，并通过可见界面操作 Wellcee 房源笔记扩展。

## 准备 Chrome

1. 在执行任何浏览器操作前，加载并遵循 `chrome:control-chrome`。
2. 明确选择用户的 Chrome 会话。不得替换成应用内浏览器、HTTP 请求、网页搜索或其他自动化方式。
3. Wellcee 流程不依赖登录状态，不得要求用户登录或等待登录。
4. 尽量复用已有的 Wellcee 标签页；没有可用页面时，在 Chrome 中打开 `https://www.wellcee.com/`。
5. 在数字 ID 的房源详情页确认扩展已显示「我的房源笔记」和收藏控件。如果控件缺失，请让用户把本仓库作为未打包扩展加载到 Chrome 并刷新页面。不得直接修改扩展存储。

浏览器 API 以 `chrome:control-chrome` 加载的当前文档为准。不要在本 Skill 中固化浏览器客户端调用方法。

## 限定操作范围

浏览前，把用户请求映射到最小授权范围：

| 用户意图 | 可执行操作 |
| --- | --- |
| 筛选、推荐、比较 | 只读取和报告，不修改房源数据 |
| 收藏 | 只收藏与用户高意愿条件匹配、达到保存门槛的房源；已收藏的房源保持不变 |
| 记笔记 | 只为本次范围内的房源新增或更新笔记 |
| 打分 | 只给已经收藏的最终 target 房源设置用户指定或按规则推导的评分；评分不得触发收藏 |
| 取消收藏、清除评分、删除笔记 | 只执行用户明确要求的破坏性操作 |

除非用户明确要求，不得联系房东、发送消息、申请、预约、付款、导出、导入或删除无关数据。

如果用户没有说明城市，且 Chrome 中也没有可用的 Wellcee 结果页，请询问城市。把「必须」「不要」「最多」「至少」等明确要求视为硬性条件；把用户明确表示「最看重」「很在意」「优先考虑」或直接决定收藏意愿的要求视为高意愿条件；其他「最好」「偏好」视为普通软性偏好。用户没有明确高意愿条件时，使用其反复强调或最直接影响是否愿意租住的已知要求，不得凭空补充。可以说明无风险的合理假设，但不得编造缺失条件。

用户没有指定结果数量时，默认以找到 5 套合格房源为目标；用户指定数量时，以用户要求的数量为目标。持续筛选直至达到目标，不得另设房源卡片数或详情页数上限。只有结果已经耗尽、可访问的合格房源数量不足或页面阻碍无法继续时，才可以提前停止；此时返回实际找到的数量并说明原因，不得为凑数降低用户的硬性条件。未完整浏览全部结果时，不得声称已经覆盖所有房源；应报告实际检查范围。

把通过硬性条件，并与用户高意愿条件匹配到值得收藏程度的最终入选房源定义为 target。浏览或比较过但未达到收藏门槛的房源不是 target。用户直接指定查看某套房源，不代表该房源自动成为 target；用户明确要求收藏该房源，或它已经收藏且用户要求评分时，才把它纳入本次 target。

## 筛选房源

1. 把请求整理成简短的硬性条件、高意愿条件、普通软性偏好和待确认项清单。
2. 使用 Wellcee 的可见筛选器设置网站支持的条件。保留不与本次要求冲突的现有筛选条件。
3. 检查房源卡片，只记录 Wellcee 页面明确显示的标题、租金、位置、户型、入住时间、房源链接，以及明确标注的最后登录或最近活跃时间等事实。
4. 打开有希望的候选房源，优先查看房源图集，再在详情页核实重要文字信息。卡片与详情页信息冲突时，以详情页为准。
5. 如果列表页或详情页明确显示「最后登录」或「最近活跃」时间，使用当前本地日期判断新鲜度；详情页时间优先。时间早于当前本地日期向前推两个自然月所得的日期时，说明「消息可能过时」，并排除出目标房源。恰好两个月不属于「超过两个月」。不得把发布时间、更新时间或入住时间当作最后登录时间；页面未提供或含义不明确时，不据此排除。
6. 排除明确违反硬性条件的房源。信息缺失的房源应单独标为待确认，不得直接视为匹配。
7. 先按硬性条件满足程度排序，再按高意愿条件的匹配程度排序，最后考虑普通软性偏好和重要取舍。
8. 在评分前独立作出收藏判断：只有高意愿条件已核实且匹配到值得保存的房源才进入 target。信息不足以确认关键高意愿条件时，标为待确认，不得仅因预期评分较高而收藏。

Wellcee 使用懒加载和前端路由。滚动、筛选或切换页面后，等待可见页面稳定并重新定位元素，不得复用失效的元素引用。优先使用无障碍名称和可见文本，避免依赖脆弱的 CSS 选择器。

依次执行浏览器写操作，避免把操作落到错误房源上。尽量复用一个结果页标签和一个详情页标签；不得关闭任务开始前已经打开的标签页。

## 重点查看房源图片

用户条件涉及装修、房间布局、采光、空间大小、家具家电、收纳、整洁度、厨卫状况、窗外环境等需要视觉判断的信息时，必须把图片作为重点检查对象：

1. 打开详情页图集，尽量查看覆盖卧室、客厅、厨房、卫生间和窗户等主要空间的图片，不要只看首图或房源卡片缩略图。
2. 围绕用户当前提出的条件观察图片。例如判断装修的新旧与风格、房间之间的连接关系、家具摆放后的可用空间、窗户和自然光线、墙面地面维护状况、厨卫设施以及可见收纳。
3. 区分图片中直接可见的事实与推断。广角镜头可能夸大空间，开灯不等于自然采光好，单张照片也不能证明隔音、朝向、面积或设施可用性。
4. 图片缺少关键空间、重复较多、清晰度过低或不同照片状况不一致时，把相关条件标为待确认，不得自行补全。
5. 留意疑似样板房线索，例如过度统一的陈设、缺少生活痕迹、不同照片的装修或户型不连贯、明显渲染图，以及与文字描述不一致。页面文字明确标注样板房、示意图或效果图时，可以直接作为依据；否则至少发现两个相互独立的可见线索，才在笔记中标注「疑似样板房」。

查看图片是为了更准确地完成用户当前的 Prompt。不要仅因为检查了图片，就强制把视觉优缺点写入评分、笔记或最终汇报；只在用户明确要求，或这些观察直接影响本次筛选结论时使用。

## 收藏 target 并评分

先完成筛选、排序和收藏判断，再确定最终 target 集合。收藏决策必须先于评分并独立成立：房源因匹配用户的高意愿条件而收藏，不得根据预计星级决定是否收藏，也不得仅因用户要求评分而收藏。确认 target 已收藏后，才分配并写入 1–5 星评分。非 target 房源可以参与内部比较，但不得写入或报告星级。

评分表示已收藏 target 集合内部的相对匹配度和质量，不是收藏门槛。用户提供评分规则时，严格采用用户规则；否则统一使用以下标准：

- `5`：收藏集合中最符合高意愿条件，整体完成度很高，且没有重要待确认风险；
- `4`：明显符合高意愿条件，整体较强，只有轻微妥协；
- `3`：已达到收藏门槛，但存在明显取舍或重要未知信息；
- `2`：只在少数高意愿点上值得保留，整体有较大缺口；
- `1`：收藏集合中的最低优先级，只因某个用户很在意的明确匹配点而保留，并存在重大取舍。

星级高低不得改变已经独立作出的收藏判断；即使 target 只有 1 星，也仍是因高意愿条件匹配而收藏。未入选的候选保持未评分状态。明确标注推断和不确定性，不得把未经核实的信息计为加分项。

## 操作扩展

从每套房源的数字 ID 详情页执行修改，确保房源身份明确。

执行每一次写笔记、收藏或评分前，都从当前地址栏的 `/rent-apartment/数字ID` 路径提取房源 ID，并确认它与目标候选的 ID 完全一致。ID 不一致时，先打开目标房源链接并重新核对；仍无法确认时跳过修改并报告，不得只根据标题、页面顺序或视觉相似度判断房源身份。

### 添加笔记

1. 编辑前读取完整的现有笔记。
2. 保留用户原有内容。除非用户明确要求替换，否则追加一条简短记录。每次 Codex 新增的记录必须独占一行；已有内容时只用一个换行符分隔，不插入空白行。
3. 如果列表页或详情页明确显示「最后登录」或「最近活跃」时间，在笔记中按页面原文写「最后登录：xxxx」；两处都有时使用详情页内容。页面未提供或含义不明确时省略该字段。
4. 查看页面可见的发布者身份标识、账号自述、房源数量和文案风格，判断是否疑似中介或管家；结合房源图片判断是否疑似样板房。页面明确标注身份或发布者自行说明时，可以直接作为依据；否则至少需要两个相互独立的可见线索，才写入对应的「疑似」判断。未达到门槛时省略该字段，不得把猜测写成事实。
5. 用户没有指定格式时，新增内容必须保持为单行，不得在字段之间换行，并使用：

   ```text
   匹配：已核实的优点；取舍：已核实的缺点；最后登录：页面显示时间；发布者：疑似中介；样板房：疑似；待确认：缺失或含糊的信息 - by Codex YYYY-MM-DD HH:mm
   ```

   发布者疑似管家时写「发布者：疑似管家」。存在明确时间时优先保留「最后登录」字段；只有存在对应疑似线索时才保留「发布者」或「样板房」字段，其他没有内容的字段可以省略。始终以 `- by Codex YYYY-MM-DD HH:mm` 结尾，其中 `HH:mm` 使用 24 小时制。日期和时间必须采用用户当前的本地时区，不得默认换算成 UTC。
6. 整条新增记录包括字段名、标点和签名在内，尽量不超过 150 字。内容过长时，优先保留与用户硬性条件直接相关的结论、最后登录时间、重要风险、疑似身份或样板房标记和时间签名，压缩或省略次要观察；不得换成多行。
7. 避免重复添加内容等价的 Codex 记录；能够明确识别时，更新相关记录。
8. 确保完整笔记不超过 2,000 个字符。不得静默截断原有内容；追加后超限时询问用户。

### 添加收藏

先记录该房源满足的高意愿条件和收藏依据。收藏依据独立成立时，才执行收藏；不得把目标星级、评分排名或评分控件需要收藏作为收藏理由。

点击前先检查控件状态。如果无障碍名称为「取消收藏此房源」，说明已经收藏，应保持不变；否则点击「收藏此房源」，并确认控件变为已收藏状态。

不得盲目切换收藏状态。取消收藏会同时删除该房源的评分。

### 设置评分

先确认当前房源属于最终 target 集合，且收藏状态已经存在并验证成功；任一条件不满足时都跳过评分。满足条件时，点击无障碍名称与目标分数一致的星级按钮，例如「4 星」。继续处理其他房源前，确认页面已显示 `4/5`。

如果用户要求给未收藏房源评分，不得为了评分自动收藏。先按高意愿条件独立判断：达到收藏门槛时，因该匹配关系收藏并验证，然后评分；未达到或无法确认时，不收藏、不评分，并说明评分的收藏前提未满足。

## 验证与恢复

每次收藏或评分后立即验证：

- 收藏：控件显示已收藏状态；
- 评分：页面显示用户要求的数值。

如果前端路由切换后控件缺失，先等待并再次检查，必要时只刷新一次当前房源页。如果仍无法确认房源身份或操作状态，停止修改该房源并报告阻碍。不得点击无法确认用途的控件。

不得检查 Chrome Cookie、本地存储、用户配置、密码或会话文件。不得绕过网站保护措施。

## 汇报结果

返回简洁结果，包含：

- 使用的筛选条件和假设；
- 检查过的房源卡片数和详情页数；
- 按名次排列的房源表格，列出标题、租金、已核实匹配点、取舍或未知信息、评分、执行的操作和 Wellcee 直达链接；
- 有助于用户判断时，说明因硬性条件或不确定性而跳过的房源；
- 所有无法完成验证的操作。

评分列只为已经收藏并完成评分的 target 房源显示 1–5 星。非 target 或未收藏房源不得显示星级，避免把内部比较结果误作已写入扩展的评分。

明确区分「已收藏」「原本已收藏」「已写入笔记」「已评分」和「未修改」。不得仅凭完成点击就声称收藏或评分成功；只有验证对应的界面状态，才能报告收藏或评分完成。
